短链服务（短链生成篇）

方案 1：长链哈希 + 截断 + 冲突处理

对长链做哈希（MD5、SHA256），取前 N 位作为短链。

  • 优点：无需自增 ID，可批量离线生成，输入相同 URL 可得到相同短链。
  • 缺点：哈希冲突不可避免，需查询数据库或缓存判断短链是否存在，安全性较低。

方案 2：数据库自增 ID + Base 62 编码

这是最容易想到的一种算法，也是本项目所使用的方案。 思路大致如下：

  1. 数据库维护一个自增 ID。
  2. 每生成一个短链，就从数据库拿到一个新的 ID。
  3. 将 ID 编码为 62 进制（数字 + 大写字母 + 小写字母）得到短链。

这个方案的优点是：

  1. 数据库实现容易。
  2. 天然保证唯一性。

可是这个方案的缺点也很明显：

  1. 短链可被预测。
  2. 单点数据库生成 ID 可能成为瓶颈。

方案 3：号段 + 线性混淆 + Base62 编码

这个方案基于方案 2。 为了解决方案 2 的两个问题，我们可以用号段方案降低单点压力，并且使用轻量级的线性混淆。

方案 4：分布式 ID + Base62 编码

这一方案基于 Snowflake 分布式 ID 生成机制，再通过 Base62 编码将二进制 ID 转换为短链码。 其优点是：

  1. 全局唯一性：Snowflake 保证不同机器、不同时间生成的 ID 唯一。
  2. 去中心化：不依赖单点数据库即可生成 ID。
  3. 低可预测性：ID 随时间增长且包含机器信息，难以预测下一个 ID。

缺点：

  1. 短链长度较长：64 位 ID 转 Base62 最长可达 11 位，在需要极短短链的场景可能略长。
  2. ID 资源浪费：生成 ID 服务并发量过低时，可能出现资源浪费。

Snowflake 算法默认生成 64 位 ID。 我们可以重新设计其 ID 结构，降低长度。 假设我们的目标 Base 62 编码最大长度为 x，其对应的二进制位数最少为 y。 y 与 x 的关系由下式给出：

y = ⌊xlog262⌋.

为了得到更短的短链码，我们可以考虑缩减 Snowflake ID 的总位数：

  1. 去除符号位。
  2. 将时间戳精度降低，从微秒级改为秒级。
  3. 减少机器数或序列号位数。

不过，ID 位数缩减会直接降低每秒可生成的唯一 ID 数量，因此我们必须在短链长度与高并发生成能力之间权衡。 如果对生成 ID 的并发性能要求很高，那我们仍然应当使用 64 位的 ID， 如果我们对短链的长度要求很高，则可以适当降低 ID 长度。
