IO 模型

在各类现代操作系统中，有五种常见的 IO 模型：

英文名 中文名
Blocking IO 阻塞 IO
Non-blocking IO 非阻塞 IO
IO multiplexing IO 多路复用
Signal-driven IO 信号驱动 IO
Asynchronous IO 异步 IO

阻塞 IO 与非阻塞 IO

阻塞 IO 的核心特点是，在执行 IO 操作时，如果数据没有准备好，调用会一直等待，直到操作完成后才返回。

IO 多路复用

IO 多路复用的核心理念，是由操作系统提供一套机制，方便单个线程同时监听多个 IO 事件。

在不同的操作系统下，IO 多路复用有不同的具体实现。 Linux 本身就提供了 select、poll、epoll 三种 IO 多路复用方案。 不论具体的实现如何，IO 多路复用总是由以下两个部分组成：

第一部分：监听的文件集合

要使用 IO 多路复用，操作系统需要知道哪些文件需要监听。 Linux 的三种 IO 多路复用方案分别使用了不同的方式来管理监听对象：

  1. select 使用位图 fd_set 存储数据结构。 fd_set 包含一个 unsigned long[] 数组，其中每个位对应一个文件描述符，容量默认为 1024。 如果 fd 被加入集合，对应位为 1，否则为 0。 调用 select 时，fd_set 作为参数传递给内核。 select 的缺点是位图大小有上限，最多只能监听 1024 个文件。
  2. poll 使用数组 pollfd 替代 fd_set，每个元素包含 fd + 关心的事件类型。 调用 poll 时，pollfd 作为参数传递给内核。 poll 相较于 select，不再有监听数量的上限。
  3. epoll 使用红黑树 struct rb_root_cached rbr 来存储监听的文件。 红黑树每个节点的类型是 struct epitem，存储了监听的 fd 和事件类型。 用户程序不直接操作红黑树，而是通过系统调用 epoll_ctl 注册文件。

第二部分：事件通知机制

事件通知机制是指内核如何告诉应用某个 fd 可读/可写。

  1. select 的核心机制是轮询，会扫描每个 fd 看是否就绪，并将就绪 fd 对应的位设置回 fd_set，最后返回就绪文件的数量。 select 可以设置为阻塞、超时阻塞、非阻塞三种模式。
  2. poll 的核心机制和 select 一致，主要区别在于数据结构的不同。 其轮询过程是遍历数组，检查每个 fd 是否就绪。
  3. epoll 的核心机制是内核维护就绪链表 + 回调通知。 用户程序调用 epoll_wait() 等待事件。 当 fd 就绪时，内核通过回调直接将 epitem 放入就绪链表，再将事件写入用户态数组 events，最后返回就绪文件数量。 epoll 采用的是真正的事件驱动模型。

IO 多路复用是一种监听 IO 事件的模型，本身不提供读写文件的能力。 通常来说，用户在获知了可读可写的事件集合之后，再去读写相应的文件。 由于文件已经就绪，用户程序不会再长时间阻塞，由此提高了效率。

所以 IO 多路复用与前面所说的阻塞 IO 和非阻塞 IO 并不是同一层级的概念，后两者是读写方式的区别。 通常来说，IO 多路复用要与非阻塞 IO 搭配使用。 如果使用阻塞 IO，程序仍然可能由于一些原因被阻塞。 例如，文件内容被其他线程读取，或者读取的长度超过了就绪数据的长度，等等。 如果线程被阻塞，就违背了使用 IO 多路复用的初衷。

不过，Linux 并不禁止 IO 多路复用与阻塞 IO 搭配使用，但这没有什么意义，只会降低效率。 并且在较新版本的 Java NIO 中，已经禁止这一搭配。

信号驱动 IO

信号驱动 IO 的核心机制，是通过操作系统发送信号来通知程序数据已就绪。

在 Linux 中开启信号驱动 IO，需要将 f_flags 的标志位 O_ASYNC 设置为开启状态：

如果数据就绪，操作系统会发送 SIGIO 信号。 通常还需要将文件设置为非阻塞 IO 模式，这是因为信号处理函数是在异步上下文执行的。 如果是阻塞 IO，信号处理函数可能会被阻塞，导致线程无法正常运行。

可以看出，信号驱动 IO 和 IO 多路复用是同一层级的概念，它们存在的目的都是为了监听 IO 事件，避免低效率的阻塞。 不过，信号驱动 IO 一般不会被使用，这主要是因为程序运行的流程很难控制。
