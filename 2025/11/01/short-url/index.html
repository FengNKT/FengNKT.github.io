短链服务

本文简要探讨短链服务（Short URL Service）的工作原理和实现思路，并提供一个具体实现：项目地址。

短链系统是一个常见的互联网基础服务，广泛用于二维码生成、短信链接、社交媒体跳转等场景。 它的核心目标是： 将冗长的原始 URL 转换为简短的访问链接，并能快速地解析还原。

  • 协议：使用 http 或 https；
  • 域名：例子中的 example.com 部分。 短链服务通常使用较短的域名，例如新浪提供的 t.cn，Bitly 提供的 bit.ly；
  • 短链码：例子中的 2Yx4Gz 部分。 短链码一般使用 Base62 编码，即由数字、小写字母、大写字母组成。 短链码越短越好，长度一般可以设计在 6-8 个字符。

从功能上看，短链服务的核心逻辑其实非常简单： 服务维护一张“短链码 → 原始链接”的映射表。 当用户访问短链时，服务根据短链码查找对应的长链并返回一个重定向响应； 如果映射不存在，则返回 404。

    B -->|是| C[返回302, 重定向至对应长链]
    B -->|否| D[返回404]

虽然原理简单，但要让系统在真实业务环境中应对高并发访问、系统故障等问题，还需要一系列工程层面的优化。

但是缓存可能出现以下几个问题：

问题 1：缓存穿透

缓存穿透是指用户请求的数据在缓存和数据库中都不存在，每次请求都会打到数据库上，造成压力。 我们可以考虑如下几个方案：

  1. 对短链码合法性进行初步校验。
  2. 使用布隆过滤器（Bloom Filter）在缓存层前进行预过滤。
  3. 对于查询不到的短链，在 Redis 中缓存一个“空值”一段时间，避免反复访问数据库。

问题 2：缓存击穿

缓存击穿是指高并发访问且缓存重建业务较复杂的热点 key 失效，无数的请求瞬间给数据库带来巨大的冲击。 对于我们的服务而言，重建 key 只涉及一次数据库查询和一次 Redis 写请求，比较简单，所以缓存击穿的风险不高。

问题 3：缓存雪崩

缓存雪崩是指在同一时段大量的缓存 key 同时失效或者 Redis 服务宕机，导致大量请求到达数据库，带来巨大压力。 对于我们的短链解析服务而言，缓存 key 的过期时间更可能是随机的，因此主要需要关注 Redis 服务宕机的问题。 可以考虑如下几个方案：

  1. 使用 Redis 集群提高可用性。
  2. 添加限流策略。
  3. 使用多级缓存。

我们在这里使用 Caffeine 提供的本地缓存。 使用方法也十分简便，只需要给函数体添加 @Cacheable 注解。

短链生成服务

短链生成服务的核心在于短链生成算法的设计。 为了构建一个合理、高效的短链生成算法，我们需要先分析短链服务所具备的主要特征：

  1. 并发特性：短链解析请求的并发量通常很高，而短链生成的并发量相对较低。
  2. 性能需求：用户对短链解析的速度要求极高，但对短链生成的延迟容忍度较高，通常可接受数秒的等待时间。
  3. 映射关系：每个短链应当唯一对应一个长链，而同一个长链则可以对应多个不同的短链。
  4. 长度要求：短链应尽可能简短，一般长度控制在 6～8 个字符之间。
  5. 安全性：短链的生成规则不应被轻易推测，以防止被恶意猜测或批量生成。

方案 1：数据库自增 ID + Base62 编码

这是最容易想到的一种算法，也是本项目所使用的方案。 思路大致如下：

  1. 数据库维护一个自增 ID。
  2. 每生成一个短链，就从数据库拿到一个新的 ID。
  3. 将 ID 编码为 62 进制（数字 + 大写字母 + 小写字母）得到短链。

这个方案的优点是：

  1. 数据库实现容易。
  2. 天然保证唯一性。

可是这个方案的缺点也很明显：

  1. 短链可被预测。
  2. 单点数据库生成 ID 可能成为瓶颈。

为了解决这两个问题，我们可以用号段方案降低单点压力，并且使用轻量级的线性混淆。

方案 2：长链哈希 + 截断 + 冲突处理

对长链做哈希（MD5、SHA256），取前 N 位作为短链。

  • 优点：无需自增 ID，可批量离线生成，输入相同 URL 可得到相同短链。
  • 缺点：哈希冲突不可避免，需查询数据库或缓存判断短链是否存在，安全性较低。

方案 3：分布式 ID + Base62 编码

类似 Snowflake 算法生成分布式 ID，再 Base62 编码。

  • 优点：保证唯一性，不依赖单点数据库，低可预测性。
  • 缺点：生成 ID 后短链长度可能较长，且短链生成服务并发量较低，可能资源浪费。

统计模块

短链服务除了生成和解析功能外，还应提供统计分析能力，例如：

  1. 访问次数、来源统计；
  2. 地理位置与设备分布；
  3. 日志采集与异常监控。

记录解析日志

由于短链解析请求频繁，若在主线程中直接记录解析日志或统计数据，会严重影响响应延迟。 因此，我们采用主线程快速响应 + 异步后台持久化的方式来记录解析日志。

实现核心思路： 1. 每次解析成功或失败后，生成一条 ResolveRecord； 2. 主线程将记录放入内存队列 ResolveRecordQueue 中； 3. 后台由一组消费者线程从队列取出记录，异步批量持久化； 4. 应用关闭时，主动清空队列，防止数据丢失。
